## 结论

- 硬性完整性：当前未发现 26人不齐、中文名缺失、俱乐部中文名缺失、球员级 `source_log` 缺失、`form_status_1_5` 缺失这类红灯项。
- 质量缺口：仍有较多球队缺身价、身高、惯用脚、逐人分钟、同源外部评分或评分缺口说明。
- 评分口径已调整：外部评分不再作为硬性必填；同场同源可核验才填。不可得时必须写 `rating_source_status` 和 `rating_missing_note`，模型主用 `form_status_1_5`、分钟、事件、伤停和角色证据。
- 内部评分口径已调整：`form_status_1_5` 与逐场 `internal_match_rating_1_5` 允许 1.0-5.0 的一位小数，并作为项目内部模型字段动态迭代。成员表应新增 `逐场内部评分记录`。
- 技术兼容：部分 JSON 带 UTF-8 BOM，PowerShell 可解析，但 Node/严格 JSON 读取会失败；后续模型脚本若使用 Node/Python 严格读取，需先去 BOM 或在读取层兼容。
- 并行影响：并行写入本身不会改变预测结果，但主线程不能直接消费 `partial`、`in_progress` 或含质量缺口未说明的文件；必须经统一验收后再进入模型。

## 处理原则

1. 预测前只允许使用已通过主线程验收的文件。
2. `partial` 球队可以参与模型，但必须对相关因子降权，并在预测输出中明示数据缺口。
3. 可补字段优先补：身价、身高、惯用脚、逐人分钟、换人、牌、同源外部评分或评分缺口说明。
4. 不可稳定核验字段不得硬编：官方逐人评分、T-75m 首发、最新训练伤停、部分小协会球员身高/惯用脚。
5. 外部评分不可得时不追求硬补；必须补齐 `rating_source_status`、`rating_missing_note`，必要时用可解释的 `internal_rating_proxy` 供模型低权重参考。
6. 复盘后必须追加逐场内部评分记录；未同步该表时，复盘或成员表验收应标记为 `partial`。
5. JSON 兼容风险应在下一轮整理时统一去 BOM，避免模型脚本读取失败。
