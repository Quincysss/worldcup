# 工作流纪律

更新时间：2026-06-10

## 1. 脚本长度与稳定性

为降低子线程中断和崩溃风险，所有线程遵守：

- 优先输出 Markdown、JSON、CSV 等数据与报告，不优先生成 Python。
- 若必须生成 Python，单文件建议不超过 300 行。
- 单个 Python 文件硬上限为 500 行。
- 超过上限必须拆分为多个小模块，例如：
  - `io.py`
  - `schemas.py`
  - `metrics.py`
  - `odds.py`
  - `poisson.py`
  - `simulation.py`
  - `reporting.py`
- 大量球队、球员、赔率、赛程数据必须放入 JSON/CSV，不得硬编码进 Python。
- 每个脚本只做一件事：采集、清洗、计算、模拟、报告输出不要塞进一个文件。
- 子线程不得为了“省事”输出几千行单体脚本。

## 1.1 持久化输出协议

子线程应按任务价值决定是否落地 Markdown/JSON/CSV 产物：需要复用、校验、回测、成员表迭代、赔率快照或长期追踪的内容必须写入本地文件；一次性解释或短结论可以直接在聊天中展开。

文件很多或结构复杂时，可以先创建草稿或骨架作为组织方式；这只是可选工程手段，不是强制步骤。

落地文件必须保留：

- `phase`
- `team` / `group` / `match`
- `status`
- `created_at` 或 `updated_at`
- `owner` 或来源线程
- `scope`
- `source_log`
- `missing_fields` / `missing_inputs`（如存在）
- `notes` 或质量说明

状态约定：

- `partial`：已有可用内容，但字段或来源不完整
- `blocked`：缺少上游文件、权限、来源或关键事实，无法继续
- `complete`：本阶段产物完成并通过必要校验

禁止行为：

- 覆盖已有有效文件而不合并
- 只在聊天里总结应当长期复用或回测的内容
- 把未完成文件伪装成 `complete`

## 2. 阶段边界

项目分为五个阶段，必须按用户明确需求进入。

| 阶段 | phase | 允许输出 | 禁止输出 |
| --- | --- | --- | --- |
| 球队档案 | `team_profile` | 单队基本信息、阵容、伤停、近期表现、战术、强度、市场事实、风险 | 比分预测、小组出线预测、冠军预测 |
| 小组横向比较 | `group_comparison` | 小组内球队强弱、风格相克、赛程环境比较 | 具体比分、投注结论 |
| 单轮/单场预测 | `match_prediction` | 指定比赛或指定轮次的比分、胜平负、大小球/盘口倾向 | 未被指定的比赛预测 |
| 小组出线预测 | `group_projection` | 小组排名、出线概率、第三名形势 | 赛事冠军预测，除非用户要求 |
| 赛事路径预测 | `tournament_projection` | 淘汰赛路径、冠军概率 | 未经小组数据支撑的跳跃式结论 |

## 3. 用户意图解释规则

当用户说“分析某队”时，只进入 `team_profile`。

例如：

- “分析挪威队” = 只做挪威球队档案。
- 不自动预测挪威小组赛比分。
- 不自动预测 I 组出线。
- 不自动给冠军路径。

当用户明确说“预测 I 组第一轮”时，才进入 `match_prediction`。

当用户明确说“预测 I 组出线”时，才进入 `group_projection`。

## 4. 推荐推进顺序

先完成一个小组内所有球队的 `team_profile`，再进入该小组预测。

I 组建议顺序：

1. 挪威
2. 法国
3. 塞内加尔
4. 伊拉克

四队档案完成后，再按顺序进入：

3. I 组第二轮比赛预测
4. I 组第三轮比赛预测

## 5. 产物命名

球队档案阶段：

```text
data\packets\teams\{team-slug}-team-packet.json
data\packets\tactics\{team-slug}-tactics-packet.json
data\outputs\team_reports\{team-slug}-model-brief.json
data\reviews\red_team\{team-slug}-red-team-review.md
队伍\{球队中文名}\球队总览.md
```

比赛预测阶段：

```text
data\packets\matches\{match-id}-match-packet.json
data\outputs\match_reports\{match-id}-prediction.json
data\reviews\red_team\{match-id}-prediction-review.md
比赛\未开始比赛\小组赛\{组名}\{match-id}.md
```

## 6. 完成定义

单队 `team_profile` 完成需要：

- 数据采集包
- 战术/教练包
- 模型输入简报
- 核验/反方审查
- 汇总可读报告

小组预测不得在该组所有球队 `team_profile` 完成前启动，除非用户明确要求“先临时预测”。

## 7. 赛后复盘联动球员表

每次做单场、单轮或小组赛后复盘时，必须同步迭代参赛球队的成员表和球员状态表。复盘不能只更新模型参数。

必须检查并更新：

- `data\outputs\player_state\{team-slug}-player-state.json`
- `队伍\{球队中文名}\成员表.md`
- 如 roster 中已有动态字段，也同步更新 `data\packets\rosters\{team-slug}-roster.json`

最低更新字段：

- 本场是否首发、出场分钟、换上/换下时间。
- 进球、助攻、黄牌、红牌、点球、乌龙、关键失误。
- 伤停、疑似伤病、停赛、累积黄牌和分钟限制。
- `form_status_1_5` 及变更原因；该字段为项目内部模型状态分，范围 1.0-5.0，允许保留小数点后一位，不要求整数。
- 每名球员每场必须追加一条 `internal_match_rating_1_5` 逐场内部评分记录；范围 1.0-5.0，允许一位小数，必须写明评分理由和模型证据。
- 赛后外部评分仅在同场同源、可核验时填写；若不可得，必须写 `rating_source_status` 和 `rating_missing_note`，不把外部评分设为硬性必填。
- 必要时可写 `internal_rating_proxy`，但必须标明它是基于分钟、进球、助攻、牌、失误、角色和状态证据推导的内部代理值。
- 影响下一轮模型的备注，例如体能、角色变化、位置变化、信心和战术职责。
- 每个时效字段的 `source_log` 和 `captured_at`。

成员表 Markdown 必须在原始 canonical 文件中原地维护：

- 唯一允许的成员表 Markdown 路径是 `队伍\{球队中文名}\成员表.md`。
- 不得为赛后、轮次或日期另建 `成员表_赛后迭代_*.md`、`成员表_第二轮赛后迭代_*.md`、`成员表_patch_*.md` 等副本。
- 已有赛后增量内容必须合并回 `成员表.md` 的 `逐场内部评分记录` 或对应球员行；合并后删除临时副本，避免模型读到过期分支。
- 若只完成关键球员更新，必须在 `player_state_update_status` 写 `partial`，并列明待补齐球员或缺失来源。

```markdown
## 逐场内部评分记录

| 比赛 | 日期 | 对手 | 球员 | 首发 | 分钟 | 事件摘要 | 外部评分/来源状态 | 内部评分1-5 | 评分理由 | 赛后状态值 | 伤停/可用性 | captured_at |
| --- | --- | --- | --- | --- | ---: | --- | --- | ---: | --- | ---: | --- | --- |
```

该表可以按比赛分段，但必须能追踪每个球员每场的内部模型评分变化。

完成复盘前必须在复盘 Markdown 或结构化 JSON 中写明 `player_state_update_status`：

- `updated`：已同步更新球员状态表和成员表。
- `partial`：只更新了部分字段，必须列出缺口。
- `blocked`：无法更新，必须说明缺少哪类来源，并把回派任务交给数据采集线程。

若 `player_state_update_status` 不是 `updated`，该复盘只能标记为 `partial`，不能标记为完整完成。

## 8. 相关线程强制参与协议

主线程不得把数据采集、战术教练、数据模型、红队核验、汇总评论等角色只在主线程内模拟执行。凡是球队分析、比赛预测、赛后复盘、竞彩建议、模型校准、成员表迭代等实质任务，必须恢复真实相关线程参与。

最低线程参与要求：

- 球队分析：数据采集 agent、战术/教练 agent、数据模型 agent、核验/反方 agent、汇总/评论员输出 agent。
- 比赛预测：数据采集 agent、战术/教练 agent、数据模型 agent、核验/反方 agent、汇总/评论员输出 agent；涉及竞彩或投注时追加竞彩/投注风险线程。
- 赛后复盘：数据采集 agent、战术/教练 agent、数据模型 agent、核验/反方 agent、汇总/评论员输出 agent；若复盘投注得失，追加竞彩/投注风险线程。
- Git 同步：只有用户明确要求同步、push、提交或迁移时才调用 Git 同步线程。

主线程职责：

- 派发前先检查目标文件是否已存在，避免覆盖有效内容。
- 派发时写清楚任务范围、目标文件、禁止项、交付格式和验证要求。
- 各线程按任务需要生成 Markdown/JSON/CSV 产物，文件骨架和小步拆分只作为可选组织方式。
- 主线程必须读取或确认每个必要线程的产物摘要后，才能把任务标为完成。
- 如果线程工具不可用、线程系统报错、或子线程长期无响应，主线程可以临时兜底，但必须在状态中标记 `partial_thread_fallback`，列明缺失线程和补救任务；不得把兜底结果伪装成完整多线程产物。

完成门槛：

- `线程状态.md` 必须追加本轮派发记录、线程 ID、产物路径、校验结果和完成/缺口状态。
- 最终聊天回复只输出预测表或复盘结论摘要、文件路径、校验结果、关键风险。
- 没有真实线程产物或明确记录的 `partial_thread_fallback`，不得写“已完成”。
